# V0.5

> 历史阶段记录：本文描述 V0.5 当时的目标和默认配置，部分模型名、preset 路径和参数已被后续版本替换。当前 live 行为以 `README.md`、`doc/API.md`、`doc/BACKEND.md`、`doc/OPERATIONS.md` 和实际配置文件为准。

## 本轮目标

`V0.5` 的重点不是继续堆功能，而是把 narrator / keeper / selector 三层真正拉开职责：

- narrator：只负责写最终 RP 正文
- keeper：负责把本轮变化写回 state，稳 continuity
- selector：负责在 narrator 开写前决定这一轮要不要补世界书、NPC 候选、NPC 档案

目标方向：

- 世界自己流转
- 主角是参与者与观察者，不是唯一驱动器
- narrator 不再被一堆同权重上下文块压成保守摘要器

## Narrator

本轮 narrator 侧的主要调整：

- `prompts/runtime-rules.md` 中明确加入“世界自主流转”规则：
  - 世界中的人物、秩序、后果和背景动作会继续运行
  - 主角不是唯一驱动器
- `backend/narrator_input.py` 中继续保留：
  - 强约束层
  - 连续性层
  - 候选知识层
- 对过渡态场景增加轻约束：
  - 不要塌成一句摘要
  - 但也不要为了“有戏”而每轮硬塞危险感

## Preset

`V0.5` 也重新定义了 preset 的职责：

- preset 不再承担世界真相、主角控制权或知情边界这类长期规则
- 这些长期规则应放在 `prompts/runtime-rules.md`
- preset 现在主要负责：
  - 节奏倾向
  - 镜头重心
  - 外部推进强弱
  - lorebook / NPC 候选的预算偏好

当前默认 preset：

- `runtime-data/default-user/presets/world-sim-core.json`

它是从旧的多份 preset 中提炼出的单一默认版本，目标是保留“有用的风格调制”，去掉与 runtime rules 重复的主规则。

旧 preset 现已归档到：

- `runtime-data/default-user/presets/legacy/`

同时，`config/runtime.json` 也已收回到最小主链用途：

- 保留真正仍在控制主链行为的 `sources / memory / model_defaults / entity_recovery / web / trace`
- 移除没有实际控制力的 `bootstrap / refresh_policy / state_policy / persona_policy`

## Keeper

本轮 keeper 侧没有推倒重做，而是继续围绕上游信号质量做收紧：

- `important_npc_tracker.py`
  - 提高泛称型人物进池门槛
  - 离场后的泛称型人物更快退出
- `thread_tracker.py`
  - `risk / clue` 不再直接挂半句对白
  - `main thread` 开始使用本地 `thread label composer`
  - 目标是把 `main thread` 从抽象词（如“暴露风险”）拉回到更可演的当前局势标签
- `skeleton keeper`
  - 当前默认从“每轮”改为“每 2 轮一次”
  - 当前默认由 `kimi-k2-0905-preview` 承担高频 skeleton / sidecar 角色
  - 首轮 bootstrap 不跑 skeleton，直接走一次完整 `state_keeper` 定底
  - 目标是在尽量保住 state 骨架稳定的前提下，减少单轮额外 keeper 耗时

- `state_keeper`
  - 当前默认由 `kimi-k2-0905-preview` 承担完整整理角色
  - 主要负责首轮定底与 12 轮 consolidation 的完整收束

当前 keeper 相关模型已统一为 `kimi-k2-0905-preview`，不再维护多模型分工。当前策略是：

- `Kimi` 负责首轮定底
- `Kimi` 负责每 2 轮 skeleton / sidecar
- `Kimi` 负责 12 轮整理

字段维护策略也进一步收紧：

- `immediate_goal` 已从 narrator 主链降级
- `main_event` 改为低频维护
- `location` 也从高频自由抽取中降级，优先依赖首轮定底与 12 轮整理

## Selector

这是 `V0.5` 的核心新增层。

当前已落地：

- 新增独立模块：`backend/selector.py`
- selector 当前是规则版，不跑额外模型
- 当前已接入控制的块：
  - `lorebook_text`
  - `system_npc_candidates`
  - `lorebook_npc_candidates`
  - `npc_profiles`

selector 的职责：

- narrator 开写前，决定这一轮是否值得补这些上下文块
- 不直接决定世界真相
- 不直接写正文

当前 selector 输入：

- `state_json`
- `recent_history`
- `keeper_records`
- `active_threads`
- `important_npcs`
- `onstage / relevant`
- `lorebook_entries`
- `system_npc_candidates`
- `lorebook_npc_candidates`

当前 selector 输出：

```json
{
  "selector_version": 1,
  "inject_lorebook_text": true,
  "inject_npc_candidates": false,
  "npc_profile_targets": ["皂衣人"]
}
```

当前已开始引入两层检索：

- 第一层：`event_summaries` 命中，判断当前是否碰到旧事件
- 第二层：只有命中时，才进一步允许长程 `summary_text` 注入 narrator

同时，selector 也开始生成一层轻量 `npc_roster`：

- narrator 现在优先吃 `npc_roster`
- 不再直接依赖多套 `onstage / relevant / important NPC` 分类
- keeper 仍然维护全量 NPC 结构，roster 只是 selector 给 narrator 的轻量视图
- 当前 `npc_roster` 已收成更现实的三字段：`name / role / status`

## 规则版条件注入

### 世界书正文

当前规则：

- 当前场景明显存在背景解释需求时注入
- 当前地点与某条 lore 有明确匹配时注入
- 当前 `thread / keeper / recent history` 已明确指向某条 lore 时注入
- 普通安顿、回屋、观察、生活流过渡段默认不注入

### 系统级 NPC / 世界书 NPC 候选

当前规则：

- `relevant_npcs` 在 recent / active_threads 中被明确提到时注入
- `important_npcs` 在 recent 中被明确提到时注入
- 候选人物名本身在 recent / active_threads / important_npcs 里真实命中时注入

### 相关 NPC 档案

当前规则：

- 默认只给 `onstage NPC`
- 再补少量在 recent / active_threads 中被明确命中的 `relevant / important NPC`

## 当前结论

`V0.5` 还不是“最终解法”，但已经完成了结构重排：

- narrator 不再单独承担“要不要看世界书/NPC 档案”的决策
- keeper 继续负责 state 稳定
- selector 开始接手上下文调度

当前最明显的收益：

- `npc_profiles` 已经开始明显减负
- narrator 开始更少吃无条件候选知识块
- `main thread` 开始从抽象风险标签回到更可演的局势标签
- `summary` 已从 `runtime.json -> sources` 主配置中降级，当前主要保留为导入兼容、trace 与历史层，不再作为 narrator 主链输入

当前仍需继续观察：

- `lorebook_text` 和 `npc_candidates` 在某些 session 中仍然较容易被触发
- `important_npcs` 仍可能保留太多泛称人物
- `main thread` 虽然已有改善，但还没总能达到最理想的“具体局势句子”

## Event Ledger

`V0.5` 还新增了一层最小 `event_ledger`：

- 模块：`backend/event_ledger.py`
- 目标：在 keeper 之前先把 narrator 正文里的“局势句”和“局部碎片”分开

当前职责：

- 输出 `main_event_candidates`
- 输出更稳定的 `main_event_candidate`
- 输出 `discarded_fragments`
- 提供 `scene_shift` 的结构化信号

当前策略：

- 不靠题材关键词过滤，而是通过 `fragment_score / scene_score` 的结构化比较来判断某句更像“局部碎片”还是“局势句”
- keeper 高频写回时，优先保留更像局势摘要的已有骨架字段，不轻易让正文局部句子覆盖 `main_event`
- 当前已升级为 `Kimi` 版 MVP：规则版 ledger 仍保留为 fallback，keeper 高频链优先使用 `Kimi ledger` 产出的 `main_event_candidate / risk_candidates / clue_candidates / scene_shift`
- `event_summary` 当前不是每轮都跑，而是默认每 3 轮一次（前两轮保留），不再额外依赖 `scene_shift` 触发
- 生成结果会落到 session-local `memory/event_summaries.json`，作为后续 selector 检索的事件索引层
- `scene_shift` 当前已收紧为结构判定：优先要求“地点切换 + 在场核心群重组”共同成立，避免普通推进轮也被误判成切段
- 当前 `event_summary` 不再追求极短摘要，而是按最近 1~3 轮如实总结，长度放宽到约 80~150 字，优先保留局势经过、参与方和后果关系

当前边界：

- `event_ledger` 只负责事件结构，不负责 NPC 过滤
- NPC 过滤链保持独立，不与 event_ledger 合并

## 后续建议

`V0.5` 之后的优先级：

1. 继续收紧 `important_npcs`
2. 继续打磨 `main thread label composer`
3. 若规则版 selector 仍偏宽，再考虑小模型 judge 作为第二阶段
