# Threadloom V0.3

## 当前方向

`v0.3` 的核心不是继续堆新的记忆层，而是把 narrator 的主输入改成：

- 滚动窗口为主
- keeper 记录为辅
- 硬状态只保留锚点
- `summary` 不再作为 narrator 主输入

## 上下文层级

### Layer 1: Rolling Window

最近 `12` 对 `user/assistant` turn 直接给 narrator。

目标：
- 保留语气
- 保留互动火候
- 保留最近的关系变化

这是当前 narrator 的第一优先级上下文。

### Layer 2: Keeper Record Archive

更早的历史不再走自由记忆 agent，而是命中以前 keeper 已经产出的结构记录。

来源文件：
- `runtime-data/<user>/characters/<character>/sessions/<session>/memory/keeper_record_archive.json`

这些记录只用于补充：
- 地点连续性
- 人物连续性
- 物件连续性
- 旧线程延续

最近 `12` 对 turn 以内的窗口记录默认不参与第二层检索，避免压过 rolling window。

archive 记录的最低要求：
- 至少有 `2` 个稳定人物
- 至少有一条 `ongoing_events` 或 `open_loops`
- 不保存明显模板化、过空或无法复用的窗口记录

## 当前 narrator 输入顺序

`v0.3` narrator 输入建议顺序：

1. `runtime-rules`
2. 硬锚点
3. 人物连续性表
4. 最近窗口
5. 活跃线程
6. 重要物件与持有关系
7. 较早结构记录
8. 相关 NPC 档案 / 世界书候选
9. 当前用户输入

优先级：

- 最近窗口高于一切软摘要和旧记录
- keeper archive 只能补充，不能覆盖最近窗口
- 硬锚点高于 recall 层
- `summary` 仍可保留为 session-local 写回 / 调试产物，但不再进入 narrator prompt
- 世界书注入当前按预算控制：只保留少量底板条目、少量命中条目，并限制单条与总注入长度
- 当前预算已细化到 `runtimeScope / entryType` 级别，避免单一类型条目吃满整个世界书预算

## 前端对应关系

当前前端不再把侧边栏当成旧式摘要面板，而是更接近 `v0.3` 的结构状态视图：

- 时间 / 地点
- 主要事件
- 在场 NPC
- 相关 NPC
- 重要 NPC
- 关键物件
- 活跃线程

消息区默认只加载最后一页历史，用户可通过“加载更早记录”继续向上翻页。

## keeper 的角色

在 `v0.3` 中，keeper 仍然是系统根源，但职责更清晰：

- keeper 负责当前结构状态
- keeper 负责生成可复用的窗口结构记录
- keeper 不再替 narrator 生成近场摘要
- 当前主链也不再依赖独立 `mid digest`

换句话说：

- narrator 负责“写得像人”
- keeper 负责“结构别乱”

## 当前 keeper 样本

当前 `v0.3` 的 keeper 调教不是围着一张卡做，而是用三种不同文本生态的样本分别压测不同问题：

- `维克托·奥古斯特`
  - 结构清楚、人物集中、场景切换明确
  - 主要用来校准：
    - `main_event`
    - `scene_core`
    - `active_threads`
    - `tracked_objects`

- `九幽大陆`
  - 没有系统级 NPC，人物主要在运行中生成
  - 主要用来校准：
    - 动态 NPC 抽取
    - 长文探险线里的人物/物件稳定性
    - keeper archive 在长记录中的延续性

- `血蚀纪`
  - 强关系、短句、高情绪、室内互动为主
  - 主要用来校准：
    - 场景锚点
    - 日常互动型文本的 `main_event`
    - 假人物过滤

当前判断：

- `维克托`：已进入“可用 + 细调”阶段
- `九幽大陆`：已进入“可用，继续扩窗口”阶段
  - 当前 `gemma-4-31b-it` 中等窗口样本里，已经能同时保住：
    - `main_event`
    - 主线程
    - 稳定人物
    - 弱语义 `role_label`
- `血蚀纪`：人物层已开始稳定，当前主问题转为场景锚点与事件标签

## keeper 测试策略

当前长记录 keeper 不再按“每轮都调模型”暴力跑，而是：

- `heuristic seed + keeper refine`
- `keeper_stride`
- `keeper_max_tokens`
- 限流等待 / 429 重试

也就是说：

- heuristics 先把骨架立住
- keeper 负责补结构质量
- 长记录默认按窗口和低频 keeper 跑

当前 keeper 模型策略：

- `qwen3.6-plus`
  - 更适合做 keeper 参考模型 / 对照模型
  - 结构质量通常更高，但额度和成本压力更大

- `gemma-4-31b-it`
  - 当前已经证明可以承担 keeper 的中等窗口长跑
  - 更适合作为日常 keeper / 长记录 keeper 的默认测试模型
  - 当前 `九幽大陆` 的长记录窗口里，人物、事件、线程都已能稳定保住
  - 非系统 NPC 的 `role_label` 也已开始从 `待确认` 进入弱语义阶段，例如：
    - `当前互动核心人物`
    - `长辈协助者`

- `Llama-3.3-70B`
  - 当前主要问题不是链路不通，而是免费层限流与 provider 策略更容易打断离线长跑
  - 目前更适合作为后续对照目标，而不是离线长跑 keeper 的唯一基线

当前实际策略：

- 用 `gemma-4-31b-it` 继续推长记录 keeper 稳定性
- 用 `qwen3.6-plus` 做参考答案和难样本对照

## 当前目标

`v0.3` 先解决三件事：

1. narrator 不再被旧摘要字段强势驱动
2. 旧历史 recall 改成命中 keeper 结构记录
3. keeper 记录的质量变成后续优化重点

## 暂不作为主路线的内容

- 独立中程摘要层
- 独立 memory agent
- 向量记忆召回
- 多层自由检索

这些可以作为后续增强，但不是当前主路线。
