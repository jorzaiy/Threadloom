# Threadloom V0.4

## 版本概述

V0.4 在 V0.3 的核心架构上新增多项功能扩展，聚焦于：用户体验完善、多用户支持、数据导入、抽取流水线升级。

## 新增功能

### 1. 默认角色卡（碎影江湖）

每个新用户自动获得碎影江湖作为默认角色卡。

- 模板目录：`runtime-data/_template/characters/碎影江湖/`
- `ensure_user_root(user_id)` 在用户首次登录时自动创建目录结构并复制模板
- 包含完整 source（character-data.json, lorebook.json, openings.json, system-npcs.json 等）
- 不包含 sessions 目录（从新游戏开始）

### 2. SillyTavern 聊天记录导入

支持从 SillyTavern 导出的 `.jsonl` 格式聊天记录导入。

**后端 API：**
- `POST /api/chat/preview` — 预校验：解析 JSONL 内容，提取角色名和消息统计，对比当前活跃角色卡
- `POST /api/chat/import` — 实际导入：验证角色名匹配后创建新会话并导入消息

**验证机制：**
- 从 JSONL metadata 和 assistant 消息中推断角色名
- 与当前活跃角色卡名称对比，不匹配则拒绝导入
- 前端显示预览信息（角色名、消息数、匹配状态）

**前端 UI：**
- 设置面板中新增「导入 SillyTavern 聊天记录」区域
- 文件选择器 (.jsonl) + 预览检查按钮 + 确认导入按钮
- 导入成功后自动切换到新会话

### 3. 多用户管理

支持多用户模式，默认关闭，由管理员手动开启。

**核心模块：** `backend/user_manager.py`

**认证方案：**
- 简单密码认证 + bcrypt 哈希存储
- 会话令牌（session token），7 天有效期
- 令牌通过 `Authorization: Bearer <token>` 或 `Cookie: session_token=<token>` 传递

**用户管理：**
- `default-user` 为管理员，不可删除
- 管理员可创建、删除普通用户，设置管理员密码
- 新用户自动执行 `ensure_user_root()` 获得默认角色卡
- 用户数据存储在 `runtime-data/_system/users.json` 和 `sessions.json`

**多用户开关：**
- `site.json` 中的 `multi_user_enabled` 字段
- 关闭时所有请求默认使用 `default-user`
- 开启后从请求头提取令牌解析用户身份

**API 端点：**
- `GET /api/auth/me` — 获取当前用户信息
- `POST /api/auth/login` — 登录，返回令牌
- `POST /api/auth/logout` — 注销
- `GET /api/users` — 用户列表（管理员）
- `POST /api/users` — 用户管理操作（create/delete/set_admin_password）
- `POST /api/multi-user` — 开关多用户模式

**前端 UI：**
- 设置面板中新增「用户管理」区域（仅管理员可见）
- 多用户开关、管理员密码设置、用户创建/删除
- 登录弹窗（多用户模式下使用）

### 4. LLM 首次设置引导

新用户或未配置 LLM 时自动引导完成设置。

- 前端初始化时检测 `siteConfig.status`
- 未配置（无 base_url 或 status 非 ready）时自动打开设置面板
- 显示引导横幅提示填写站点地址和密钥
- 配置保存成功后引导自动关闭

### 5. 物品 / 情报抽取流水线升级

将物品和情报抽取从 LLM 直出改为三阶段流水线：

```
启发式提候选 → LLM 判定/分类/去重 → 本地 merge
```

**物品抽取（object_bootstrap_agent.py）：**
- 启发式：四策略并行提取
  1. 动作动词 + 已知物件名词（高置信度）
  2. 动作动词 + 短名词（需过滤）
  3. 量词 + 已知物件名词（无需动词前缀）
  4. 直接匹配已知多字物件名词（≥2字）
- 已知名词库按长度降序排列，保证长词优先匹配（如 "钥匙牌" > "钥匙"）
- 垃圾过滤：虚词字符、方位/时间后缀、非便携物（家具/地名/人称）
- LLM：判定是否为可持续追踪物件，标准化 label/holder/state/notes
- Merge：与 `object_registry.json` 合并，去重、更新持有者状态
- 注册表：`memory/object_registry.json`（含版本、处理进度、物件列表）

**情报抽取（clue_bootstrap_agent.py）：**
- 启发式：信息传递/发现/秘密/证据/组织等触发模式匹配
- LLM：分类类型（hidden/rumor/evidence/intel）和置信度
- Merge：与 `clue_registry.json` 合并，去重、更新置信度
- 注册表：`memory/clue_registry.json`（含版本、处理进度、线索列表）

**事件抽取：** 暂不升级（与 arbiter 创建流程冲突，保持现有模式）

**LLM 回复解析：**
- 三个 bootstrap agent 统一使用 `_strip_code_fences()` 处理 LLM 返回的 markdown 代码围栏（` ```json ... ``` `）
- `state_keeper_candidate` 的 `max_output_tokens` 已从 280 提升至 800，避免 JSON 输出截断

**测试结果（碎影江湖 87 轮对话）：**
- NPC: 5 实体，100% LLM 分类（陆小环、深衣青年、镇北司皂衣人、栖云客栈掌柜、栖云客栈小二）
- 物品: 12 件，100% LLM 分类（风灯、布巾、外袍、铁牌、短刀、血衣、伤药等）
- 情报: 12 条，100% LLM 分类（intel/evidence/hidden 三类，含置信度）

**调用位置：**
- `context_builder.py`：构建上下文时触发增量抽取
- `keeper_archive.py`：归档时触发增量抽取

## 安全加固（审计修复）

### 后端

- **原子写入**：`user_manager.py` 和三个 bootstrap agent 的 save 函数全部改用 `tempfile + os.replace` 原子写入，防止并发/崩溃导致 JSON 文件损坏
- **认证加强**：多用户模式下管理员必须设置密码，不再允许空密码登录绕过
- **日志记录**：密码验证失败、registry 加载异常、LLM 降级等场景均添加 `logging.warning/error`，便于排查
- **异常隔离**：`keeper_archive.py` 中 bootstrap 调用包裹 try/except，单个 registry 失败不影响整体流程

### 前端

- **XSS 防护**：所有 `innerHTML` 中包含动态数据的位置（状态栏、聊天导入预览、用户列表）全部改为 `textContent` + DOM 操作
- **登出功能**：新增登出按钮（设置 → 用户管理），调用 `POST /api/auth/logout` 并清除本地 token
- **401 自动登录**：`apiJson` 检测 401 响应自动弹出登录窗；`checkAuth` 失败时同样触发

## 与 V0.3 的关系

V0.4 不修改 V0.3 的核心架构（滚动窗口 + keeper archive + 硬状态锚点），仅在此基础上：
- 扩展了抽取流水线（从 NPC 扩展到物品和情报）
- 增加了用户层面的管理能力
- 补充了数据导入通道
- 完善了新用户体验
